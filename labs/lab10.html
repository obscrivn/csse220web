01. Inheritance Submission

Step 1: Understand the Design

 Read First

Before writing code, think about the difference between inheritance and interfaces.

Use inheritance when:

  • One class is a more specific version of another (is-a)
  • Classes share data (fields) and behavior (methods)

Use an interface when:

  • Different classes share a capability (can-do)
  • They do NOT need to share fields or common implementation

Examples in this lab:

  • Drink extends MenuItem because a drink is a menu item
  • Smoothie extends Drink because a smoothie is a drink
  • Discountable is an interface because an item can be discounted

Think: Should this relationship share data? Or just behavior?

Step 2: Make Drink a Subclass of MenuItem

A Drink is a kind of MenuItem, so this relationship should use inheritance.

Your tasks:

  • Update Drink so it extends MenuItem.
  • Complete the full constructor so it calls the superclass constructor with super(...).
  • Initialize the iced field in Drink.
  • Add a method public boolean isIced() that returns whether the drink is iced. Use the exact method name isIced().

Note: In Java, boolean methods typically use is instead of get. For example: isEmpty(), isEnabled().

💡 Hint
  • Use inheritance when one class is a more specific version of another class.
  • A Drink should reuse the shared data from MenuItem instead of redefining it.
  • If the parent class has a constructor, the subclass must call it.

Step 3: Use this(...) to Chain Constructors

The Drink class has a shorter constructor that should not repeat setup code. Instead, it should call another constructor in the same class.

Your tasks:

  • Complete the shorter Drink constructor using this(...).
  • Use the shorter constructor to make drinks iced by default.
💡 Hint
  • Use this(...) when you want one constructor to call another constructor in the same class.
  • This helps reduce duplicated initialization code.
  • this(...) must be the first line in the constructor.

Step 4: Make Smoothie a Subclass of Drink

A Smoothie is a more specific kind of Drink. It should inherit the drink data and behavior, then add its own field.

Your tasks:

  • Update Smoothie so it extends Drink.
  • Add and initialize the flavor field.
  • Complete the constructor using super(...) to initialize inherited data.
💡 Hint
  • Use super(...) when the subclass needs the parent class to initialize its part of the object.
  • A Smoothie should not repeat the setup code that already belongs in Drink or MenuItem.
  • super(...) must be the first line in the constructor.

Step 5: Override getDescription()

The MenuItem class already provides a description. A smoothie should keep that base description and then add more specific information.

Your tasks:

  • Override getDescription() in Smoothie.
  • Call super.getDescription() to reuse the parent version.
  • Add the smoothie flavor to the final returned string.
  • super.getDescription() returns something like: "Smoothie ($5.0)"
  • Your final result should look like: "Smoothie ($5.0) - Strawberry"
  • Use string concatenation with +.
💡 Hint
  • Use overriding when a subclass needs the same method name but more specific behavior.
  • Use super.method() when you want to keep the parent behavior and add to it.
  • This is better than rewriting the whole description from scratch.

Step 6: Implement the Discountable Interface

Some menu items can provide a discount. That is a shared capability, so this relationship should use an interface.

The Discountable interface defines the following method: getDiscountAmount() that return double

Your tasks:

  • Update Smoothie so it implements Discountable.
  • Complete the required interface method.
  • Return a fixed discount amount of 1.50.
💡 Hint
  • Use an interface when classes share a behavior or capability, but do not need to share fields or constructors.
  • Discountable is about what an object can do, not what it is.
  • Very different kinds of items could be discountable.

Step 7: Run the Tests

Run the provided JUnit tests as you work. The tests will help you check both correctness and design.

Before moving on, make sure:

  • Drink extends MenuItem.
  • Drink uses both super(...) and this(...) correctly.
  • Smoothie extends Drink.
  • Smoothie overrides getDescription().
  • Smoothie uses super.getDescription().
  • Smoothie implements Discountable.
  • All tests pass.
02. Books Refactoring (Part 2) Submission

Lab-Design 1: Fix the Broken System

In this lab, you will analyze a poorly designed Java system and redesign it using better object-oriented design ideas. This is not just about making code work. It is about learning how to recognize design problems and improve them.

Scenario Context

A website tracks books and the kids who read them.

  • Each Book has a name and author
  • Each Kid has a name and grade level
  • The teacher records when a kid reads a book

The system must support:

  • Printing all kids who read a given book
  • Printing all books read by a given kid

The current system works, but its design has serious issues. Your job is to analyze and redesign it.

Day 2 Analysis

Step 1: Understand the Current Design

Before changing anything, review the provided Books and Kids code.

  • What classes exist?
  • Who owns the data?
  • Where do responsibilities feel unclear?

Team Task: Spend 5–7 minutes discussing before coding.

Step 2: Decide What to Fix First

You should NOT start coding immediately. As a team, decide your first refactoring step.

  • What is the most important problem to fix?
  • Why does it matter?
  • What will you change first?

Write down your decision before coding.

Step 3: Refactor the Design

  • Make small, incremental changes
  • Test frequently
  • Do NOT rewrite everything at once

Focus on:

  • Clear class responsibilities
  • Reducing duplication
  • Improving data ownership

Step 4: Team Roles

  • Driver: writes code
  • Navigator: checks design decisions
  • Reviewer: asks “why are we doing this?”
