01. Broken UML Submission

Resources 1: Access & Scope

💻 Coding guide: Fix the Broken Code

A broken export removed important Java design information from a game project. Your job is to restore the correct design by choosing the proper:

  • access modifiers: private, default, protected, public
  • static fields
  • final constants
  • correct use of scope
  • fix the bug in setHealth so that it correctly updates the player's health

Step 1: Scenario + UML

In your game, each Player stores information about its own health, level, and score. Some of this data should be carefully protected, while some should be shared across all players.

Your team received a broken UML/code export where the access modifiers and some keywords were lost. Your job is to restore the correct design based on the intended behavior below. 


@startuml
title Broken UML Export - Player Design

interface Healable {
    heal(amount : int) : void
}
class Player {
    health : int
    level : int
    score : int
    totalPlayers : int
    MAX_HEALTH : int

    setHealth(health : int) : void
    getHealth() : int
    addScore(points : int) : void
    heal(amount : int) : void
}
Player ..|> Healable
@enduml

Design Requirements

  • Each player has its own health. Health belongs to one specific player and should not be changed directly by other classes. Other code should use methods to update or read it safely.
  • Each player has its own level. Level belongs to one specific player and should be stored as part of that player's internal state.
  • Each player has a score. Score belongs to one specific player and should be visible to subclasses, because special kinds of players may need to update or use score directly later.
  • The game tracks the total number of players. This value is shared across all players, so there should be only one copy for the whole class.
  • The game has a maximum health value. This value is the same for every player, should be shared by all players, and should not be changed after it is defined.
  • A player can be healed. Healing is a behavior required by the Healable interface.
  • When setting health, the code should correctly update the player’s field. Avoid scope mistakes such as assigning a parameter to itself.

  • which fields should be final

02. Books Refactoring (Part 1) Submission

Lab-Design 1: Fix the Broken System

In this lab, you will analyze a poorly designed Java system and redesign it using better object-oriented design ideas. This is not just about making code work. It is about learning how to recognize design problems and improve them.

Overview Design Refactoring

You are given:

  • a scenario
  • a bad UML design
  • a bad Java implementation in the bad package

Your team will study the bad design, identify what is wrong, and create a better version in a new package called good.

This activity will take place over two class days.

Scenario Context

A website tracks books and the kids who read them.

  • Each Book has a name and author
  • Each Kid has a name and grade level
  • The teacher records when a kid reads a book

The system must support:

  • Printing all kids who read a given book
  • Printing all books read by a given kid

The current system works, but its design has serious issues. Your job is to analyze and redesign it.

Team Roles Groups of 3

Work in a team of 3 students. Each person should have one role.

  • Navigator - leads discussion and keeps the team focused on design decisions
  • Driver - explores code and implements changes when coding begins
  • Recorder - writes down design problems, team decisions, and reflections

You will rotate roles on Day 2.

Day 1: Understand and Redesign Analysis

Step 1: Understand the Current System

Run the program and study the UML and code in the bad package.

As a team, discuss these questions:

  • Where are books stored?
  • Where are kids stored?
  • How does the system find a book?
  • What happens when multiple kids read the same book?

Step 2: Identify Design Problems

Find at least 3 important design problems in the current system.

Use these prompts to guide your thinking:

  • Is any data duplicated?
  • Is anything hard or inefficient to find?
  • Is any class doing too much work?
  • Is ownership of data clear?

The Recorder should use the provided google doc to create one for team and write down your team’s answers.

Step 3: Redesign the System

Create a better design that solves the main problems.

Your redesign should:

  • avoid duplicate books
  • make lookup easier and cleaner
  • clearly define ownership of data
  • reduce unnecessary searching and coupling
