01. Multiple Buttons

In this lab we will create a reusable MyCounterClass that extends JPanel, displays a label, a read-only text field, and three buttons, and updates the counter when the buttons are clicked.

New focus: instead of using a lambda for addActionListener, you will implement a named inner class that implements ActionListener and overrides actionPerformed.

1. Create the Counter Panel

  1. Define public class MyCounterClass extends JPanel
  2. Declare your components and state: 
    private JLabel    label;
private JTextField text;
private JButton    up, down, reset;
private int       count = 0;
  3. In the constructor:
    • Instantiate fields:
      label = new JLabel("Counter");
      text = new JTextField(count+"",10); text.setEditable(false);
      up = new JButton("Count Up");
      down = new JButton("Count Down");
      reset = new JButton("Reset");
    • Set your layout and preferred size: 
      setLayout(new FlowLayout());
setPreferredSize(new Dimension(300, 60));
    • Add them in order: 
      add(label);
add(text);
add(up);
add(down);
add(reset);

2. Implement the Named Listener Class

Inside MyCounterClass, define a private inner class: 

private class ButtonListener implements ActionListener {
  @Override
  public void actionPerformed(ActionEvent e) {
    String buttonLabel = e.getActionCommand();
    if (buttonLabel.equals("Count Up")) {
      count++;
    } else if (buttonLabel.equals("Count Down")) {
      count--;
    } else {  // "Reset"
      count = 0;
    }
    text.setText(count + "");
  }
}

Then add this Listener to each button: 

up.addActionListener(new ButtonListener());
down.addActionListener(new ButtonListener());
reset.addActionListener(new ButtonListener());

3. Compare with Lambda-Based Listener

Previously we have done this: 

up.addActionListener(e -> myMethod());
public void myMethod() {
  count++;
  text.setText(count + "");
};

Pros of named inner class:
• Centralizes all three button behaviors in one actionPerformed
• Easier to add logging or shared preprocessing
• Good practice for more complex event handling
Cons: Slightly more boilerplate than a single lambda

4. Testing

  1. Click each button and verify:
    • Count Up increments
    • Count Down decrements
    • Reset sets back to zero

5. Reflection

  • Why must the inner class implement ActionListener and override actionPerformed?
  • How does getActionCommand() let you distinguish which button fired the event?
  • When would you choose a named listener class over a lambda (and vice versa)?

Part 1. Custom Drawing

Focus: overriding paintComponent() and calling repaint() to update the drawing.

1. Define MyCanvas as a JPanel

public class MyCanvas extends JPanel {
          public static final int WIDTH  = 400;
          public static final int HEIGHT = 140;
          public static final Color BG   = Color.CYAN;
          public static final Color FG   = Color.BLACK;
      
          private int x1 = WIDTH/2, x2 = WIDTH/2;
          private int y1 = HEIGHT/8, y2 = HEIGHT/8*7;
      
          public MyCanvas() {
            setPreferredSize(new Dimension(WIDTH, HEIGHT));
          }
      
          @Override
          protected void paintComponent(Graphics g) {
            super.paintComponent(g);           
            setBackground(BG);                 
            g.setColor(FG);                    
            g.drawLine(x1, y1, x2, y2);
          }
      
          public void moveDelta(int dx) {
            x1 += dx; x2 += dx;
            repaint();                         
          }
        }

Key takeaway: whenever you change state (x1/x2), call repaint() so Swing will invoke paintComponent() and reflect your changes.

2. Control Panel & KeyListener (ControlPanel)

In ControlPanel you will connect buttons and arrow-key events to your canvas panel

Define the panel and store references (see the image for specifications):

public class ControlPanel extends JPanel {
    private final MyCanvas canvas;
    private final JFrame parentFrame;
    private final JButton left, right;

Constructor

The ControlPanel needs to tell the canvas when to move the line, and it needs to re–focus the frame after each click so arrow-keys keep working. That is why its constructor takes MyCanvas canvas and JFrame parentFrame as parameters: 

public ControlPanel(MyCanvas canvas, JFrame parentFrame)
  • canvas: button and key handlers can call canvas.moveDelta(...)‌
  • parentFrame: after a button click we call parentFrame.requestFocusInWindow() and restore keyboard focus 

    public ControlPanel(MyCanvas canvas, JFrame parentFrame) {
      this.canvas = canvas;
      this.parentFrame = parentFrame;
      setLayout(new FlowLayout());

Create & Add Buttons

Instantiate your buttons and add them to this panel:

left  = new JButton("<- Left");
    right = new JButton("Right ->");
    
    add(left);
    add(right);

Button listeners

Use lambdas that call helper methods:

    left.addActionListener(e -> moveLeft());
    right.addActionListener(e -> moveRight());

Implement helper methods

  private void moveLeft() {
    canvas.moveDelta(-10);
    parentFrame.requestFocusInWindow();
  }

  private void moveRight() {
    canvas.moveDelta(+10);
    parentFrame.requestFocusInWindow();
  }

Add a KeyListener to the frame

Register an arrow-key listener on the frame:

    parentFrame.addKeyListener(new KeyAdapter() {
      @Override
      public void keyPressed(KeyEvent e) {
        switch (e.getKeyCode()) {
          case KeyEvent.VK_LEFT:  canvas.moveDelta(-10); break;
          case KeyEvent.VK_RIGHT: canvas.moveDelta(+10); break;
        }
      }
    });

Why?

  • addActionListener handles mouse clicks on buttons
  • KeyListener captures LEFT/RIGHT arrow keys when the frame has focus

4. Reflection Questions

  • Why must we call repaint() inside moveDelta()? What happens if we update the coordinates without calling repaint()?
  • Constructor Dependencies: Why does ControlPanel take both the MyCanvas and the JFrame as parameters? How does this help with loose coupling and single-responsibility?
  • KeyListener vs. ActionListener: What kinds of events does each one handle? Why do we need a KeyListener in addition to the button ActionListeners?
  • Focus Management: Why is requestFocusInWindow() called after each button press? What would happen if we omitted that call?
  • Separation of Concerns: If you wanted to add a “speed” slider (JSlider) to control how far the line moves per key press, which class would you modify, and how would you pass that new value into your drawing logic?
