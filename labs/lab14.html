02. Prep: Build the First Game Object Final Project

In this prep, you will create the first real part of your game model: a Player. The goal is to understand where game data belongs.

Big idea: GameModel owns the game state. Player owns its own position. 

MainApp
  creates GameModel and GameWindow once at startup

GameModel
  owns game objects and game rules

Player
  stores player position

GameComponent
  draws the game and handles input

Step 1: Create the Player Class

In the model package, create a new class named Player.

The player should store its position using two fields:

  • row: vertical position
  • col: horizontal position

We will also use a tile size to convert grid position into screen pixels. 

private static final int TILE_SIZE = 40;

Add:

  • private fields for row and col
  • a constructor that receives the starting row and column
  • getter methods for row and column
  • a constant TILE_SIZE
  • private fields for row and col
  • a constructor that receives the starting row and column
  • getter methods for row and column

Now add a method to draw the player: 

public void drawOn(Graphics2D g2) {
  int x = this.col * TILE_SIZE;
  int y = this.row * TILE_SIZE;

  g2.fillRect(x, y, TILE_SIZE, TILE_SIZE);
}

This converts grid position (row, col) into screen position (x, y).

Step 2: Add Movement to Player

Add a method to Player that changes the position by a row amount and a column amount. 

moveBy(int dRow, int dCol)

Examples:

  • Move up: row changes by -1
  • Move down: row changes by +1
  • Move left: column changes by -1
  • Move right: column changes by +1

This is the first piece of game logic: changing the player’s position.

Step 3: Put Player inside GameModel

Open GameModel. Add a private Player field.

In the GameModel constructor, create one player at a starting location. For example, row 5 and column 5.

Add a getter so other classes can ask the model where the player is.

Step 4: Add Movement Methods to GameModel

Add methods to GameModel for moving the player:

  • movePlayerUp()
  • movePlayerDown()
  • movePlayerLeft()
  • movePlayerRight()

Each method should call player.moveBy(...).

Why do this? Later, you can refactor and use these methods for walls, boundaries, gems, zombies, and levels before allowing movement.

Step 5: Draw Player in GameComponent

In GameComponent, call: 

model.getPlayer().drawOn(g2);

You should now see a rectangle player on the screen.

Step 6: Connect to the UI Later

Later, GameComponent will listen for keyboard input. When the user presses a key, the component should call a method on GameModel. 

Key press → GameComponent → GameModel → Player

Example idea: 

W key → model.movePlayerUp()
A key → model.movePlayerLeft()
S key → model.movePlayerDown()
D key → model.movePlayerRight()

After moving the player, the UI will repaint the screen.

Future Game Logic Roadmap

After the player works, you will gradually add more model classes.

  • Walls: prevent the player from moving into blocked tiles
  • Grid or Level: store the map layout
  • Gems: track collectibles
  • Zombies: move enemies slowly/randomly
  • Lives and Score: track game progress
  • Exit: allow level completion after collecting required items

Keep the same design idea: the model owns rules, the UI shows what happened.
