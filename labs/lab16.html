01. Abstract and Interface

In this section you’ll combine an interface and an abstract class to see how they divide responsibilities between capability (interface) and structure (abstract class).

Step 1. Create an Interface

public interface Renderable {
    void render();
}

Step 2. Create an Abstract Class Entity that Implements the Interface

public abstract class Entity implements Renderable {
    private String id;

    public Entity(String id) { this.id = id; }

    public String getId() { return id; }

    @Override
    public void render() {
        System.out.println("Rendering " + id);
    }

    public abstract void update();
}
💡 Why abstract and interface together?

Renderable defines a capability (something that can be rendered). Entity provides shared state (id) and partial implementation (render()). Subclasses only supply the details by overriding update().

Step 3. Create a Subclass Player

public class Player extends Entity {
    public Player(String id) { super(id); }

    @Override
    public void update() {
        System.out.println("Updating " + getId());
    }
}

Step 4. Test in main()

public class Demo {
    public static void main(String[] args) {
        Entity p = new Player("Alice");
        p.render();
        p.update();
    }
}
💡 Expected Output 
Rendering Alice
Updating Alice
💡 Reflection Question

Why can the variable be declared as Entity instead of Player? What OOP concept does that demonstrate?

02. Polymorphism

Now we’ll see how Java decides which version of a method to run when a superclass reference points to different subclass objects.

In this lab, you will work with a simple class hierarchy to see how Java chooses which print() method to call at runtime (polymorphism), how upcasts let you treat a subclass as its superclass, and how downcasts let you recover subclass‐specific behavior, sometimes safely and sometimes resulting in errors if misused. Note that @Override annotation is optional but strongly recommended whenever you intend to override a superclass or interface method.

🧩 Step 1. Set Up a Simple Hierarchy (PolymorphismDemo.java)

we are using inner classes A,B,C,D inside PolymorphismDemo class

public class PolymorphismDemo {
    public static void main(String[] args) {
        A x = new D();
        x.print();               // (1)
        D x2 = (D)x;
        x2.print();              // (2)
        x2.printC();             // (3)
        B x3 = (B)x;
        // x3.printC();         // (4) 
    }
}

class A {
    public void print() { System.out.println("A"); }
}

class B extends A {
    @Override public void print() { System.out.println("B"); }
}

class C extends B {
    @Override public void print() { System.out.println("C"); }
    public void printC() { print(); }
}

class D extends C {
    @Override public void print() { System.out.println("D"); }
}

Key Takeaways

  • Interfaces define what classes can do → capabilities.
  • Abstract classes define how things partly work → shared code + state.
  • Polymorphism lets one reference call different implementations at run-time.
  • Upcasting is automatic; downcasting requires an explicit cast and careful checks.
