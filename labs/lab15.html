01. Multiple Buttons

In this lab we will create a reusable MyCounterClass that extends JPanel, displays a label, a read-only text field, and three buttons, and updates the counter when the buttons are clicked.

New focus: instead of using a lambda for addActionListener, you will implement a named inner class that implements ActionListener and overrides actionPerformed.

2. Implement the Named Listener Class

Inside MyCounterClass, define a private inner class: 

private class ButtonListener implements ActionListener {
  @Override
  public void actionPerformed(ActionEvent e) {
    String buttonLabel = e.getActionCommand();
    if (buttonLabel.equals("Count Up")) {
      count++;
    } else if (buttonLabel.equals("Count Down")) {
      count--;
    } else {  // "Reset"
      count = 0;
    }
    text.setText(count + "");
  }
}

Then add this Listener to each button: 

up.addActionListener(new ButtonListener());
down.addActionListener(new ButtonListener());
reset.addActionListener(new ButtonListener());

3. Compare with Lambda-Based Listener

Previously we have done this: 

up.addActionListener(e -> myMethod());
public void myMethod() {
  count++;
  text.setText(count + "");
};

Pros of named inner class:
‚Ä¢ Centralizes all three button behaviors in one actionPerformed
‚Ä¢ Easier to add logging or shared preprocessing
‚Ä¢ Good practice for more complex event handling
Cons: Slightly more boilerplate than a single lambda

4. Testing

  1. Click each button and verify:
    • Count Up increments
    • Count Down decrements
    • Reset sets back to zero

5. Reflection

  • Why must the inner class implement ActionListener and override actionPerformed?
  • How does getActionCommand() let you distinguish which button fired the event?
  • When would you choose a named listener class over a lambda (and vice versa)?

üí° EDT (Event Dispatch Thread)

Launching Swing UIs with SwingUtilities.invokeLater ensures all UI work happens on the EDT (safe & responsive).

üí° Why invokeLater?

All Swing UI work must happen on the Event Dispatch Thread. invokeLater schedules startup there.

üí° Why call requestFocusInWindow() after button clicks?

Clicking a button gives it the keyboard focus (it ‚Äúholds the mic‚Äù). Calling requestFocusInWindow() hands the mic back to the panel so it hears arrow keys again.

üí° Difference Between add() and setContentPane()

add() adds a component (like a JPanel or JButton) inside the container‚Äôs layout. It‚Äôs used when you want multiple UI pieces to coexist in the same window, for example, buttons, labels, or nested panels.

setContentPane() replaces the entire content area of a JFrame (or another top-level container) with a new panel. Use it when you want to swap out the main view completely, e.g. switching game screens.

üß† Think of it like this:
add() ‚Üí "Put this inside the existing layout."
setContentPane() ‚Üí "Replace everything inside with this new panel."

üí° New switch Syntax for Panels (No break Needed)

In newer versions of Java (since Java 14+), you can use the enhanced switch expression syntax. Instead of writing multiple case labels followed by statements and break lines, you can use -> to make it cleaner and safer. 


switch (panelName) {
    case "menu" -> showMenuPanel();
    case "game" -> showGamePanel();
    case "settings" -> showSettingsPanel();
    default -> showErrorPanel();
}

‚ú® No need for break statements: each arrow (->) automatically ends that branch. It also works nicely with short expressions or even multi-line blocks: 


switch (panelName) {
    case "menu" -> {
        setContentPane(menuPanel);
        revalidate();
    }
    case "game" -> {
        setContentPane(gamePanel);
        revalidate();
    }
    default -> System.out.println("Unknown panel!");
}

public class GameComponent extends JComponent {
          public static final int WIDTH  = 400;
          public static final int HEIGHT = 140;
          public static final Color BG   = Color.CYAN;
          public static final Color FG   = Color.BLACK;
      
          private int x1 = WIDTH/2, x2 = WIDTH/2;
          private int y1 = HEIGHT/8, y2 = HEIGHT/8*7;

          public GameComponent() {
            setPreferredSize(new Dimension(WIDTH, HEIGHT));
          }
      
          @Override
          protected void paintComponent(Graphics g) {
            super.paintComponent(g);           
            Graphics2D g2 = (Graphics2D) g;
            g.setColor(FG);                    
            g.drawLine(x1, y1, x2, y2);
          }
      
          public void moveDelta(int dx) {
            x1 += dx; x2 += dx;
            repaint();                         
          }

     public void setX(int i) {
		x1 = i;
		x2 = i;
	    repaint();
	}
        }
üí° Why repaint()?

repaint() schedules Swing to call paintComponent(Graphics). That‚Äôs the correct way to reflect updated positions on screen (no manual loops).
