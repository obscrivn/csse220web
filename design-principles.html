Use this checklist in order and stop at the first big failure.

Rule of thumb: The first No is usually the primary violation. Later problems are often consequences.
Main Focus and Sequence (how to check principles in order)
  1. Principle #1: Functionality (can it work at all?)
  2. Principle #2: Structure around data (nouns → classes, encapsulation)
  3. Principle #3: Distribute responsibility efficiently (high cohesion)
  4. Principle #4: Minimize dependencies (low coupling: tell-don’t-ask, avoid chains)
  5. Principle #5: Avoid duplication (DRY - don't repeat yourself)
If #1 is No: the design is broken. Don’t over-analyze #2–#5 — it can’t meet the requirements.
1️⃣ Functionality: Does the design support the required functionality?

What this principle is asking: “Could this system technically meet the requirements?”
(Not “is it elegant?” Not “is it good OO?” Just: can it work?)

Guiding questions:

  • Is all required data represented somewhere in the design?
  • Are all required relationships possible (1–1, 1–many, many–many)?
  • Can all required operations be implemented without changing the requirements?
  • Even if messy, could it run and produce correct results?
Yes → move to #2.
No ❌ → broken design (everything after is basically irrelevant).
Common “#1 = No” examples
  • No place to store required entities (e.g., “teams” exist in the story but no Team structure).
  • Missing required relationships (e.g., a Department cannot contain Teams).
  • Missing required operation that can’t be added without redesign.
2️⃣ Structure around the data (Encapsulation): Is the design built from the problem nouns?

What this principle is asking: “Did you model the real things in the problem, and do they own their behavior?”

Guiding questions:

  • 2a (Nouns → classes):What are the key nouns in the prompt? Are they represented as classes?
  • 2b (Intelligent behavior):Does each class have intelligent behavior that operates on its own data?
  • 2b (Data class check):Are you avoiding data classes (only fields + getters/setters, no real behavior)?
  • 2b (Ownership of decisions):When you see a domain question (ex: “Is the team over budget?”), does the answer live in the class that owns the data?
Primary failure pattern: A “manager/controller” class does the real work while domain classes just store data. That’s usually a 2️⃣ violation first.
Quick self-check: “Where should this method live?”
  • If the method uses mostly Team data, it should probably be in Team.
  • If the method uses mostly Department data, it should probably be in Department.
  • If the method uses “a little from everyone,” that’s a red flag.
3️⃣ Distribute responsibility efficiently (High Cohesion): Is the workload split cleanly?

What this principle is asking: “Does each class have one main job, or is one class doing everything?”

Guiding questions:

  • 3a: Is any class getting too large or doing too much?
  • 3b: Does each class have a single responsibility (high cohesion)?
  • Would one requirement change force edits in many unrelated methods in the same class?
Common consequence of failing #2: One class becomes a “calculator + organizer + printer + decider.” That’s a #3 problem (low cohesion).
Red flags
  • Method names in one class cover unrelated tasks: “compute…”, “handle input…”, “print…”, “filter…”, “load…”.
  • One class knows the details of every other class.
4️⃣ Minimize dependencies (Low Coupling): Are objects “telling” instead of “asking”?

What this principle is asking: “Are classes too entangled? Are you reaching through objects to get work done?”

Guiding questions:

  • 4a (Tell, Don’t Ask): Are you pulling data out with getters and making decisions elsewhere?
  • Could you replace “ask for internals + compute” with a single message like team.isOverBudget()?
  • 4b (Message chains): Do you have chains like a.getB().getC().doThing() (train wreck)?
  • Does one class need deep knowledge of another class’s internal structure?
Fast test: If your solution needs lots of getters to work, you’re probably violating #4.
What “Tell, Don’t Ask” looks like (mini example)

Bad: Department asks Team for employees/equipment and computes the budget status itself.

Better: Department asks team.isOverBudget() and Team handles the details internally.

5️⃣ Avoid duplication: Is code duplicated?

What this principle is asking: “Will you copy-paste the same logic in multiple places?”

Guiding questions:

  • Are the same logic patterns likely to be copy-pasted across multiple classes or methods?
  • If you fixed a bug in one place, would you need to fix it again elsewhere?
  • Are multiple classes implementing nearly identical behavior?
  • Could shared logic be pulled into:
    • a helper method?
    • a common interface?
    • a superclass?
Note: Duplication is often implementation-level, not just UML-level.
Acceptable phrasing: “Code duplication cannot be fully evaluated at the UML level, but the design structure suggests potential duplication.”
How to write a full-credit design critique answer (student template)
  1. State the first principle that fails (usually #2 for bad OO designs).
  2. Explain what is misplaced (which class owns the data vs. where the behavior was placed).
  3. Name 1–2 consequences (often #3 and #4) and explain why they happen.
  4. Give a fix direction (what methods move where; who should “own” the decision).
Example sentence frame:
“This design violates 2b because behavior that belongs in X is placed in Y. This causes a 3b issue because Y becomes responsible for too many tasks, and a 4a issue because Y must ask for internals instead of telling X to do the work.”
