Working with Files

Part 1: Adding Images (Sprites) in Java: Option A vs Option B (and why getResource() is better)

If your sprite image is null, it’s usually a path/classpath issue. We will show two correct ways to organize images and load them with getResource.

✅ What getResource actually does

  • It does NOT read from your hard drive path. It reads from the classpath (the stuff Eclipse copies into your build output).
  • Think: “Find this file inside my project build,” not “Find this file in Finder/Explorer.”

Two modes:

  • SomeClass.class.getResource("file.png") → looks relative to the package of SomeClass
  • SomeClass.class.getResource("/images/file.png") → looks from the classpath root (absolute path)

Option A (Beginner-friendly): Put the image in the SAME folder as the class

Use this when you want the image to “live next to” the class. This is what we did in the Ball example. 

// Folder layout (Eclipse plain project)
src/
  model/
    Ball.java
    tennis.png   ← SAME folder as Ball.java
  ui/
    GamePanel.java

Code: (package-relative path: no leading slash)

private static BufferedImage sprite;
private static boolean triedLoad = false;

private static void loadSpriteOnce() {
  if (triedLoad) return;
  triedLoad = true;

  try {
    // tennis.png must be in the SAME package/folder as Ball.java
    sprite = ImageIO.read(Ball.class.getResource("tennis.png"));
  } catch (IOException | IllegalArgumentException ex) {
    sprite = null; // (for debugging labs, printing the error is better)
  }
}

When Option A breaks: If you move Ball to a different package (folder) but forget to move tennis.png with it, the resource becomes null.

Option B (Best for multi-package projects): Put all images in one shared folder

Use this when you have packages like ui, model, util and you don’t want “where is my png??” chaos. 

// Folder layout (recommended for team projects)
src/
  images/
    tennis.png
    wall.png
    zombie.png
  model/
    Ball.java
  ui/
    GamePanel.java

Code: (absolute path: leading slash)

private static void loadSpriteOnce() {
  if (triedLoad) return;
  triedLoad = true;

  try {
    sprite = ImageIO.read(Ball.class.getResource("/images/tennis.png"));
  } catch (IOException | IllegalArgumentException ex) {
    sprite = null;
  }
}

Why the leading / matters: It tells Java “start at the classpath root,” so this works no matter which package calls it.

How to create an images folder in Eclipse (and make Java actually see it)

  1. Create the folder: Right-click your project → New → Folder → name it images (or put it under src as src/images).
  2. Add your PNG files: drag into the folder, or Right-click folder → Import…
  3. IMPORTANT: Make sure the folder is on the Build Path so resources get copied into the build output:
    • Right-click the images folder → Build Path → Use as Source Folder

Reality check: A file can exist in your project tree and still be invisible to getResource if it’s not on the classpath.

Debug checklist (aka “Why is my image null?”)

Paste this once and run:

URL url = Ball.class.getResource("/images/tennis.png"); // or "tennis.png" for Option A
System.out.println("Sprite URL = " + url);
  • If it prints null → your path is wrong OR the folder isn’t on the classpath.
  • If it prints something like file:/.../bin/images/tennis.png → resource is found ✅

Extra pro tips (save future you)

  • Don’t load images inside paintComponent — load once, then draw many times.
  • Tiles/walls should share images (load wall.png once, reuse it for 200 walls).
  • Sprites can own their image (Ball/Player/Zombie can load their sprite once).
Part 2: Drawing “Non-Entities” (Walls, Floors, Backgrounds) — No Wall Class Needed

Some things in your game are not “entities.” They don’t move, don’t think, and don’t have unique behavior. Examples: walls, floor tiles, background, decorations. For these, creating a class per tile is usually unnecessary and makes your code harder (and slower) for no reason.

What counts as a “non-entity”?

  • Static environment: walls, floors, grass, water, path tiles
  • Background-only visuals: trees, torches, signs that do nothing
  • Map layout: where things exist (grid-based), not objects that act

Rule of thumb: If it has no behavior (no meaningful methods), it’s probably not a class yet.

Recommended approach: store non-entities as data, draw them with a loop

You typically represent the environment in one of these two ways:

Approach What you store (Model) What you do (UI) Best for
Grid / Tile Map char[][] or int[][] Double loop (row/col) and draw tiles Mazes, maps from level1.txt
Rectangles List List<Rectangle> Loop through rectangles and draw Collision-first design

Option 1: Grid / Tile Map (most common for maze levels)

Level file example (each character is one tile):

WWWWWWWWWW
W....W...W
W.WW.W.W.W
W....W...W
WWWWWWWWWW

Model idea: store the grid

// In your Level class
private char[][] grid;

public char tileAt(int row, int col) { return grid[row][col]; }
public int rows() { return grid.length; }
public int cols() { return grid[0].length; }

UI idea: load images once, then draw with nested loops

// In GamePanel (or your UI class)
private BufferedImage wallImg;
private BufferedImage floorImg;
private static final int TILE_SIZE = 32;

public GamePanel() {
  // Load ONCE (Option B style: from /images folder)
  try {
    wallImg = ImageIO.read(getClass().getResource("/images/wall.png"));
    floorImg = ImageIO.read(getClass().getResource("/images/floor.png"));
  } catch (IOException | IllegalArgumentException ex) {
    wallImg = null;
    floorImg = null;
  }
}

@Override
protected void paintComponent(Graphics g) {
  super.paintComponent(g);
  Graphics2D g2 = (Graphics2D) g;

  for (int r = 0; r < level.rows(); r++) {
    for (int c = 0; c < level.cols(); c++) {
      int x = c * TILE_SIZE;
      int y = r * TILE_SIZE;

      char tile = level.tileAt(r, c);

      if (tile == 'W') {
        if (wallImg != null) g2.drawImage(wallImg, x, y, TILE_SIZE, TILE_SIZE, null);
      } else {
        if (floorImg != null) g2.drawImage(floorImg, x, y, TILE_SIZE, TILE_SIZE, null);
      }
    }
  }
}

Performance tip: Load images once in the constructor (or a loader method), not inside paintComponent.

Option 2: List of Rectangles (great when collision is the main goal)

Model idea: build wall rectangles once after reading the grid

// In Level class
private final List<Rectangle> walls = new ArrayList<>();

public List<Rectangle> getWalls() { return walls; }

public void rebuildWallsFromGrid(int tileSize) {
  walls.clear();
  for (int r = 0; r < rows(); r++) {
    for (int c = 0; c < cols(); c++) {
      if (tileAt(r, c) == 'W') {
        walls.add(new Rectangle(c * tileSize, r * tileSize, tileSize, tileSize));
      }
    }
  }
}

UI idea: draw walls by looping the rectangles

for (Rectangle w : level.getWalls()) {
  if (wallImg != null) g2.drawImage(wallImg, w.x, w.y, w.width, w.height, null);
}

Why we usually do NOT make a Wall class (yet)

  • You might have 200+ wall tiles — that’s 200+ objects for no real benefit.
  • Walls usually have no behavior besides “block movement.” That’s not enough to justify a class.
  • It’s easier to debug a grid/list than a pile of tiny objects.

When a Wall class DOES make sense: breakable walls, doors, moving walls, damaging walls, or special interactions.

Debug checklist (if your tiles don’t draw)

  • Is your image path correct? Try printing the URL: 
    System.out.println(getClass().getResource("/images/wall.png"));
    If it prints null, Java can’t find the file on the classpath.
  • Is your images folder on the Build Path? Right-click folder → Build Path → Use as Source Folder
  • Are your tile coordinates correct? x = col * TILE_SIZE, y = row * TILE_SIZE
